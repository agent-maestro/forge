# DO-254 hardware certification — radar artifact mapping

Forge's radar vertical maps onto DO-254 (RTCA, *Design Assurance
Guidance for Airborne Electronic Hardware*) at every artifact
the auditor expects. This document is the cross-reference an
audit team uses to demonstrate compliance.

## Why DO-254 cares about radar firmware

Modern airborne radars run their pulse compression, Doppler
processing, and target tracking on FPGAs that fall under DO-254
Design Assurance Level (DAL) A or B. The guidance requires:

- A documented **algorithmic specification** prior to RTL
- **Bit-exact equivalence** between the spec and the deployed FPGA
- Coverage evidence for **requirements-based verification**
- **Configuration management** for every artifact, including
  intermediate forms

Forge produces every one of those artifacts from a single .eml
source.

## Tool qualification

Forge itself qualifies under DO-330 as a Tool Qualification Level
(TQL) 5 development tool: Forge's outputs are checked against
independent runtimes (Lean, Python reference, C reference) at
build time. The cross-target equivalence harness produces the
DO-330 §6.2 verification artifact for every covered function.

## Audit checklist

For an auditor reviewing a radar build:

1. Verify the `.eml` source has `@verify(lean, theorem = ...)`
   on every safety-critical function.
2. Run `python tools/cli/main.py <file>.eml --target lean` and
   check the produced `.lean` builds with `lake build`.
3. Run `python tools/cli/main.py <file>.eml --target verilog`
   and check the produced `.v` against the reference.
4. Run the cross-target equivalence harness and verify
   `overall: MATCH` on every covered function.
5. Run `--profile-only` and verify chain-order matches the
   declared `where` clause on each function.

Every step has CI evidence in `.github/workflows/`.
