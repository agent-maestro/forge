# ISO 26262 Compliance Guide — Monogate Forge

> ISO 26262 is the functional-safety standard for automotive
> electronics. It defines four Automotive Safety Integrity Levels
> (ASIL A through D) plus QM (quality-managed, no special
> requirements). EV powertrain control loops typically fall in
> ASIL-C or ASIL-D.
>
> This guide maps the artifacts the EML-lang forge produces
> onto the ISO 26262 evidence categories. Sister doc to
> `industries/aerospace/certification/DO_178C.md` -- the
> per-artifact mapping is similar; the per-ASIL stringency
> differs.

---

## Quick map: forge artifact → ISO 26262 evidence

| ISO 26262 evidence (Part 6: Software) | Forge artifact |
|---------------------------------------|----------------|
| **6.4.2 Safety Requirements at SW level** | `@verify(lean, ...)` clauses + `requires` / `ensures` on each function signature |
| **6.5.4 Safe Subset of Programming Language** | EML-lang's chain-order types are the language-level safety constraint; no pointer arithmetic, no aliasing, no UB |
| **6.6.4 Detailed Software Design** | The function body in `.eml` (declarative; no procedural noise) |
| **6.7.4 Software Unit Implementation** | `eml-compile <fn>.eml --target c` — gcc-compilable C99 |
| **6.7.4.4 Software Unit Verification (analysis)** | `eml-compile <fn>.eml --target lean` — formal proof of `requires → ensures` machine-checked by `lake build` |
| **6.7.4.4 Software Unit Verification (test)** | `--fpga-sim` (when verilator is on PATH) — empirical proof that the C output and the deployable HDL agree bit-for-bit |
| **6.8 Software Integration and Verification** | `--target all` produces every artifact from one source; integration is just linking the generated C |
| **8.4 Configuration Management** | The `.eml` source is the *single* source of truth for both the binary and the bitstream; no hand-translated copies |

## Per-ASIL guidance

ISO 26262 ASIL applies a multiplier to the rigor of every clause.
Forge's defaults align with ASIL-C out of the box; ASIL-D needs
tighter `where` constraints.

See `ASIL_mapping.md` for the chain-order ↔ ASIL mapping rule
of thumb (defended empirically against E-193 numerical-stability
data).

### ASIL D (highest -- catastrophic failure, e.g. unintended motor rotation)

- **Required:** `@verify(lean, ...)` annotation on every
  function in the powertrain control hot path.
- **Required:** `where chain_order <= 2` on every signal that
  feeds the inverter (no transcendental nesting).
- **Required:** `--fpga-sim` PASS gate in CI (Verilator
  bit-equivalence with the C reference).
- **Required:** `clamp(...)` on every actuator command
  (validated by an `ensures abs(result) < V_LIMIT`-style clause).

### ASIL C (high -- severe injury possible)

- `@verify(lean, ...)` on the top-level step function.
- `where chain_order <= 2` on the actuator command.
- The `motor_foc.eml` example in this directory targets
  ASIL-C out of the box.

### ASIL B (medium -- light/moderate injury possible)

- `@verify(...)` on the safety-critical functions only.
- `where chain_order <= 3` permitted.
- `clamp(...)` on actuator outputs still required.

### ASIL A (lowest -- harm possible but limited)

- `@verify(...)` optional; `clamp(...)` recommended on actuator
  outputs.

### QM (no automotive safety effect)

- All forge defaults; no special constraints.

## Worked example: `motor_foc.eml` → ASIL-C evidence

Source: `industries/automotive/powertrain/motor_foc.eml`

```eml
@target(fpga, clock_mhz = 100, precision = float32)
@verify(lean, theorem = "vd_command_within_inverter_limits")
fn foc_d_axis(
    i_d_setpoint: Real,
    i_d_measured: Real,
    i_d_integral: Real,
) -> Real
    requires (abs(i_d_setpoint) < 500.0)
    requires (abs(i_d_measured) < 500.0)
    requires (abs(i_d_integral) < 100.0)
    ensures  (abs(result) < V_LIMIT)
{ ... }
```

V_LIMIT = `400.0 / sqrt(3) = 230.94` V — the maximum phase voltage
the inverter can synthesize from a 400V DC bus. Any output above
that is unrealizable; the `ensures` clause + clamp prove the
controller never asks for it.

Run the full pipeline:

```bash
$ eml-compile motor_foc.eml --profile-only
# foc_d_axis: chain_order: 0  cost_class: p0-d4-w0-c0
# park:       chain_order: 2  cost_class: p2-d5-w2-c0  (sin/cos)
# clarke:     chain_order: 0  cost_class: p0-d4-w0-c0
# pi_step:    chain_order: 0  cost_class: p0-d2-w0-c0

$ eml-compile motor_foc.eml --target all -o ./out
# Generates motor_foc.{c,rs,lean,v} -- one source, four artifacts.

$ # Move the Lean theorem into monogate-lean for verification.
$ cp out/motor_foc.lean ../monogate-lean/MonogateEML/MotorFOC.lean
$ cd ../monogate-lean && lake build MonogateEML.MotorFOC
```

What you submit to the safety-case auditor:

| File | ISO 26262 §       | Purpose |
|------|------------------|---------|
| `motor_foc.eml` | 6.6.4 | Detailed Software Design (single source of truth) |
| `motor_foc.c`   | 6.7.4 | Software Unit Implementation |
| `motor_foc` (binary) | 6.7.4 | Object Code |
| `motor_foc.v`   | 6.7.4 | Hardware Unit Implementation (alt deployment) |
| `motor_foc.lean`| 6.7.4.4 | Software Unit Verification (analysis) |
| `--fpga-sim` MATCH | 6.7.4.4 | Software Unit Verification (test) |
| `--profile-only` dashboard | 6.4.2 | Safety Requirements traceability |

All seven derive from one `.eml`. No hand-translated copies.

## What forge does NOT (yet) do for ISO 26262

- **Tool Confidence Level (TCL) qualification.** ISO 26262
  Part 8 §11 demands tool qualification for any tool whose
  output is part of the safety case. The forge's qualification
  package is on the roadmap; today it relies on its 190-test
  CI gate as evidence.
- **Failure Modes and Effects Analysis (FMEA).** Tool-orthogonal;
  the forge produces the source the FMEA reasons over.
- **Diagnostic Coverage (DC) calculation.** External tools
  (e.g. Tessy) compute DC from the generated C.

These are scoped out because they're toolchain-orthogonal —
the forge's job is to produce the source-of-truth artifacts
ISO 26262 demands, not to replace the entire safety toolchain.
