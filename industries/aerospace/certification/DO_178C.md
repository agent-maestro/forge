# DO-178C Compliance Guide — Monogate Forge

> Aerospace software certification under DO-178C (formally
> "Software Considerations in Airborne Systems and Equipment
> Certification") demands documented traceability from
> requirements → design → source → object code, with formal
> verification of safety-critical properties.
>
> This guide maps the artifacts the EML-lang forge produces
> onto the DO-178C evidence categories, so you can use it as
> the foundation of a certified-software submission.

---

## Quick map: forge artifact → DO-178C evidence

| DO-178C evidence | Forge artifact |
|------------------|----------------|
| **High-Level Requirements (HLR)** | `@verify(lean, theorem = ...)` clauses + `requires` / `ensures` clauses on the function signature |
| **Low-Level Requirements (LLR)** | The function body in `.eml` (declarative; no procedural noise) + `where chain_order <= N` constraints (numerical-stability floor) |
| **Source Code** | `eml-compile <fn>.eml --target c` — gcc-compilable C99 with per-function profile comments |
| **Object Code** | `gcc -O2 -Wall -Werror <fn>.c libmonogate.c -lm` — produces a deterministic, single-translation-unit binary suitable for the cert audit |
| **Verification: Reviews** | The `--profile-only` CLI dumps the chain-order, cost class, dynamics counter, FPGA estimate, and any stability warnings — directly readable by reviewers |
| **Verification: Analysis** | `--target lean` emits a Lean 4 theorem (`MonogateEML.Tactics`-aware) that proves `requires → ensures`. `lake build` machine-checks it. |
| **Verification: Test** | `--fpga-sim` (when verilator is on PATH) compares the C output bit-for-bit against the generated Verilog — empirical proof that the certified C and the deployable HDL agree. |
| **Configuration Management** | The `.eml` source is the *single* source of truth for both the binary and the bitstream; no hand-translated copies to keep in sync. |

## Per-DAL guidance

DO-178C applies five Design Assurance Levels (DAL A through E).
Forge's defaults align with DAL B (typical for autopilot) out of
the box; DAL A and C/D/E need different `where` constraints.

### DAL A (catastrophic failure condition)

- **Required:** every `@target(fpga)` function MUST carry a
  `@verify(lean, ...)` annotation with both `requires` and
  `ensures` clauses.
- **Required:** `where chain_order <= 2` on every signal that
  feeds an actuator (no transcendental nesting in the
  control-loop hot path).
- **Required:** `--fpga-sim` PASS gate in CI (Verilator
  bit-equivalence against the C reference).

### DAL B (hazardous failure condition) — `autopilot.eml` example

- `@verify(lean, ...)` on the top-level step function, NOT
  necessarily every helper.
- `where chain_order <= 2` on the actuator command (already
  enforced by the `StableSignal` type alias).
- `clamp(...)` on every output that drives a physical actuator
  — caught by the `ensures abs(result) < ELEVATOR_MAX` clause.

### DAL C (major)

- `@verify(...)` only on the safety-critical functions; the rest
  pass through review-only.
- `where chain_order <= 3` permitted (no fp16-drift gate).

### DAL D (minor)

- `@verify(...)` optional. `clamp(...)` still required on actuator outputs.

### DAL E (no safety effect)

- All forge defaults; no special constraints.

## Worked example: `autopilot.eml` → cert package

Source: `industries/aerospace/flight_control/autopilot.eml`

```eml
@target(fpga, clock_mhz = 100, precision = float32)
@verify(lean, theorem = "autopilot_command_within_limits")
fn autopilot_step(
    pitch_setpoint: Real,
    pitch_measured: Real,
    pitch_integral: Real,
) -> Real
    requires (abs(pitch_setpoint) < 1.5708)   // |pi/2|, +/- 90 deg
    requires (abs(pitch_measured) < 1.5708)
    requires (abs(pitch_integral) < INTEGRAL_LIMIT)
    ensures  (abs(result) < ELEVATOR_MAX)
{ ... }
```

Run the full pipeline:

```bash
$ eml-compile autopilot.eml --profile-only
# ... per-function profile dashboard ...
# autopilot_step: chain_order: 0    cost_class: p0-d6-w0-c0
# rate_controller: chain_order: 0    cost_class: p0-d2-w0-c0
# gravity_compensation: chain_order: 2 (cos)    cost_class: p2-d4-w2-c0

$ eml-compile autopilot.eml --target c -o autopilot.c
$ eml-compile autopilot.eml --target verilog -o autopilot.v
$ eml-compile autopilot.eml --target lean -o autopilot.lean

$ # Verify SW = HW bit-equivalence (when verilator is installed)
$ pytest hardware/simulation/tests/ -v

$ # Verify the Lean theorem
$ cp autopilot.lean ../monogate-lean/MonogateEML/Autopilot.lean
$ cd ../monogate-lean && lake build MonogateEML.Autopilot
```

What you submit to the certifier:

| File | Purpose |
|------|---------|
| `autopilot.eml` | LLR + HLR (single source of truth) |
| `autopilot.c` | Source Code |
| `autopilot` (binary) | Object Code |
| `autopilot.v` | Hardware Source (alt deployment target) |
| `autopilot.lean` | Verification Analysis (formal proof) |
| Verilator harness output | Verification Test (HW=SW equivalence) |
| `--profile-only` dashboard | Reviews artifact |

All seven derive from one `.eml`. No hand-translated copies.

## Lean theorem templates

Forge ships templates for the three most common DO-178C theorem
shapes:

- `lean_templates/precision_bound.lean.j2` — bound on output
  precision over a domain
- `lean_templates/domain_safety.lean.j2` — preconditions + 
  closure (e.g. `arg > 0` for `ln`)
- `lean_templates/overflow_check.lean.j2` — output stays within
  representable range

The forge's Lean backend uses these when generating proof
skeletons; manually-written proofs follow the same shape.

## What forge does NOT (yet) do

- Stack-usage analysis (`gcc -fstack-usage` is the recommended
  external tool).
- Worst-case execution time (WCET). The Verilog cycle count is
  a precise proxy for the FPGA path; for the C path, use
  `aiT` or similar.
- Coverage tracing (MC/DC). The forge's tests are pytest-based;
  port the test harness to your DO-178C-approved coverage
  framework.

These are scoped out because they're toolchain-orthogonal —
the forge's job is to produce the source-of-truth artifacts
DO-178C demands, not to replace the entire cert toolchain.
