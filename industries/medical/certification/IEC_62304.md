# IEC 62304 Compliance Guide — Monogate Forge

> IEC 62304 is the medical-device software lifecycle standard.
> It defines three software safety classes: A (no injury possible),
> B (non-serious injury possible), C (death or serious injury
> possible). FDA 510(k) submissions cite IEC 62304 evidence as
> their software-quality basis.
>
> This guide maps forge artifacts to IEC 62304 evidence
> categories. Sister doc to the aerospace DO-178C and automotive
> ISO 26262 guides; same forge pipeline, different evidence
> shape.

---

## Quick map: forge artifact → IEC 62304 evidence

| IEC 62304 § | Artifact |
|-------------|----------|
| **5.2** Software requirements analysis | `@verify(lean, ...)` clauses + `requires` / `ensures` on each function |
| **5.3** Architectural design | The `.eml` module structure + type aliases (chain-order constraints) |
| **5.4** Detailed design | Function bodies in `.eml` (declarative, no procedural noise) |
| **5.5** Implementation | `eml-compile <fn>.eml --target c` |
| **5.6** Integration + test | `--target all` produces every artifact from one source; integration is just gcc-linking the generated C |
| **5.7** System testing | `--fpga-sim` (when verilator is on PATH) bit-equivalence of HW vs SW |
| **6.1** Maintenance plan | The `.eml` source IS the source of truth; binaries / bitstreams regenerate from it |
| **7.1** Risk management (ISO 14971) integration | The `@verify` `ensures` clause is the safety-control evidence |

## Per-class guidance

### Class C (death or serious injury possible) — `infusion_pump.eml` example

- **Required:** `@verify(lean, ...)` on every function in the
  delivery hot path.
- **Required:** explicit `requires` clauses for every input
  (FDA reviewers want to see the assumed input domain).
- **Required:** `ensures` on the actuator command (motor
  speed, valve position, etc.) gating it within physical
  limits.
- **Required:** `where chain_order <= 2` on actuator outputs
  (no fp16 drift in the safety-critical path).

### Class B (non-serious injury possible)

- `@verify(...)` on safety-affecting functions only.
- `where chain_order <= 3` permitted.
- `clamp(...)` on all actuator outputs.

### Class A (no injury possible)

- `@verify(...)` optional. Forge defaults are fine.
- Mostly used for telemetry / display / logging code.

## Worked example: `infusion_pump.eml` → 510(k) evidence

Source: `industries/medical/devices/infusion_pump.eml`

```eml
@target(fpga, clock_mhz = 50, precision = float32)
@verify(lean, theorem = "infusion_motor_command_safe")
fn motor_command(
    prescribed_rate: Real,
    measured_rate: Real,
    rate_integral: Real,
) -> Real
    requires (RATE_MIN <= prescribed_rate)
    requires (prescribed_rate <= RATE_MAX)
    requires (abs(measured_rate) < RATE_MAX)
    requires (abs(rate_integral) < 100.0)
    ensures  (MOTOR_MIN <= result)
    ensures  (result <= MOTOR_MAX)
{ ... }
```

The `ensures (MOTOR_MIN <= result <= MOTOR_MAX)` is the
machine-checkable evidence that the motor command CANNOT
exceed the physical actuator's safe operating range — even
if the prescribed rate / sensor reading / integral term go
to their requires-clause boundaries.

```bash
$ eml-compile infusion_pump.eml --target all -o ./out
# Produces motor_command.{c,rs,lean,v} -- one source, four artifacts
$ # Move the .lean into monogate-lean for verification
$ cp out/infusion_pump.lean ../monogate-lean/MonogateEML/InfusionPump.lean
$ cd ../monogate-lean && lake build MonogateEML.InfusionPump
```

What you submit to the FDA reviewer:

| File | IEC 62304 § | Purpose |
|------|------------|---------|
| `infusion_pump.eml` | 5.4 | Detailed Design (single source of truth) |
| `infusion_pump.c` | 5.5 | Implementation |
| `infusion_pump` (binary) | 5.5 | Object Code |
| `infusion_pump.lean` | 5.7 | System Testing (analysis) |
| `--fpga-sim` MATCH | 5.6 | Integration + test (HW = SW) |
| `--profile-only` dashboard | 5.2 | Requirements traceability |
