# FIPS 140-3 Compliance Guide — Monogate Forge

> FIPS 140-3 (Federal Information Processing Standard 140-3,
> "Security Requirements for Cryptographic Modules") is the US
> government's certification regime for cryptographic modules.
> Approval is required for any crypto module sold into US
> federal agencies and most regulated industries (healthcare,
> finance, critical infrastructure).
>
> This guide maps the artifacts the EML-lang forge produces
> onto the FIPS 140-3 evidence categories so you can use it as
> the foundation of a Cryptographic Module Validation Program
> (CMVP) submission.

---

## Quick map: forge artifact → FIPS 140-3 evidence

| FIPS 140-3 evidence | Forge artifact |
|---------------------|----------------|
| **Approved algorithms (FIPS 197 AES, FIPS 180-4 SHA-256, FIPS 186-5 ECDSA, FIPS 203 ML-KEM, ...)** | `industries/crypto/symmetric/<algo>.eml` etc. — every `.eml` carries a `@verify(lean, theorem = "<algo>_correct")` attestation |
| **Algorithm correctness proofs (CAVP)** | `eml-compile <algo>.eml --target lean` emits a Lean 4 theorem with the FIPS test-vector input/output as the hypothesis/conclusion |
| **Module specification** | The function signature + `requires` / `ensures` clauses are the formal module spec; `--profile-only` dumps a human-readable summary |
| **Side-channel resistance evidence** | `ConstantTime` type checker output: every approved-algorithm function is typed `ConstantTime`, which mechanically forbids chain-order-1+ ops and key-dependent branches |
| **Cryptographic Algorithm Self-Test (CAST)** | `eml-compile --target c` emits a power-up self-test harness driven by the FIPS test-vector file; built into the deployable binary |
| **Source code review** | The `.eml` source is small (typically 30-80 lines per primitive) and declarative; reviewers trace `requires` → body → `ensures` without procedural noise |
| **Object code traceability** | `--target verilog` emits a single Verilog file per `.eml`; the bitstream's gate count matches the SuperBEST-optimized node count, providing a checkable mapping |
| **Approved random bit generator boundary** | RNGs are explicitly *not* FIPS-approved here; forge uses an `extern` boundary calling the host's FIPS-approved DRBG (e.g., NIST SP 800-90A CTR_DRBG) |

## Per-security-level guidance

FIPS 140-3 has four security levels (1 = software, 4 = tamper-
resistant hardware with EM-emanation defense). Forge artifacts
unblock evidence at all four:

### Level 1 (software)
  - Approved algorithm bodies (`.eml`) compile to constant-time
    C99 + Lean theorems. Sufficient for software-module
    submission with documented host-OS boundary.

### Level 2 (tamper-evidence)
  - Same as Level 1 plus a tamper-evidence enclosure for the
    host. Forge contributes nothing new at this layer (it's a
    physical packaging requirement).

### Level 3 (tamper-resistance + identity-based authentication)
  - Forge's FPGA target shines here: the bitstream is
    deterministic (same `.eml` → same gate placement), so the
    physical-attack assumptions of the cert apply uniformly.

### Level 4 (tamper-detection + EM resistance)
  - Constant-time-by-type-check is necessary but not sufficient.
    Forge documents the *timing* side channel; *power* and *EM*
    side channels still require dedicated countermeasures
    (masking, hiding) layered on top of the chain-order-0 core.

## Worked example: AES-256 SubBytes (the headline FIPS 197 primitive)

```bash
# Generate FIPS 197 test-vector harness
eml-compile industries/crypto/symmetric/aes.eml \
    --target c --emit-fips-cast > aes_cast.c

# Generate Lean theorem for CAVP submission
eml-compile industries/crypto/symmetric/aes.eml \
    --target lean > AesCorrect.lean
lake build       # machine-checks the theorem

# Generate constant-time-resistant Verilog
eml-compile industries/crypto/symmetric/aes.eml \
    --target verilog --constant-time-check strict > aes_sbox.v

# Side-channel-equivalence smoke test (C vs Verilog)
eml-compile industries/crypto/symmetric/aes.eml \
    --fpga-sim    # bit-for-bit cmp against verilator
```

Each step's output is a separate piece of CMVP evidence; the
total auditable surface area is much smaller than the
equivalent hand-written submission because every artifact is
derived from the same `.eml` source rather than maintained as a
parallel copy.

## What forge does NOT cover (yet)

  - **Approved random-bit generators (SP 800-90A).** RNGs are
    side-effectful and live outside the EML-lang functional
    purity model. Forge integrates with a host-provided DRBG
    via an `extern` boundary; the DRBG itself must be separately
    FIPS-validated.
  - **Key-management (SP 800-57).** Key derivation, storage,
    and zeroization are runtime concerns handled by the host
    integrator, not by the algorithm-implementation `.eml`.
  - **Physical-security testing.** Level 3+ requires
    tamper-evidence and tamper-resistance physical tests outside
    the scope of any compiler.

## CMVP submission checklist

| Step | Owner | Forge contribution |
|------|-------|--------------------|
| 1. Module specification | integrator | `eml-compile --emit-spec` |
| 2. Approved algorithms list | integrator | `industries/crypto/*/index.json` |
| 3. CAVP algorithm validation | integrator | per-algo Lean theorems |
| 4. Module security policy | integrator | `--profile-only` dumps |
| 5. Self-test harness | integrator | `--emit-fips-cast` |
| 6. Side-channel analysis | integrator + lab | `ConstantTime` type-check audit log |
| 7. Operational testing | lab | (forge contributes test vectors) |

The forge column lists the artifact each `eml-compile`
invocation produces. Steps 1-5 are entirely automatable from
the `.eml` source; steps 6-7 are the human-in-the-loop pieces
where the lab ties the forge output to the physical cert
requirements.
