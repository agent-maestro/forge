# Software targets — compile to C, Rust, Python, LLVM, WASM

> Five software backends, one source. Pick the one that fits your
> deployment runtime; every target produces output that's
> bit-equivalent within 1e-12 of the SymPy reference for any
> EML expression in the arithmetic subset.

---

## At a glance

| Target | Output                                  | Toolchain after compile        |
|--------|-----------------------------------------|--------------------------------|
| `c`    | C99 source linking `libmonogate.h`      | `gcc -O2 -Wall -Werror`        |
| `rust` | Rust source consuming `monogate-sys`    | `cargo build`, `cargo clippy`  |
| `python` | Python module using `math.*`          | `python3` (no third-party deps) |
| `llvm` | Portable LLVM IR text                   | `llc` / `clang` for native     |
| `wasm` | wasm32 bytecode (or LLVM IR fallback)   | `wasm-tools`, browser, Node.js |

Quick look at numerical equivalence:

```
$ pytest tests/equivalence/test_cross_target.py -v
```

The harness compiles every demo through every available toolchain
and asserts ULP agreement.

## C backend (`--target c`)

```
$ eml-compile sigmoid.eml --target c -o sigmoid.c
$ gcc -O2 -Wall -Werror sigmoid.c \
      software/runtime/c/libmonogate.c \
      -I software/runtime/c -lm -o sigmoid.out
```

Output is C99 — no GNU extensions, no C11 atomics, no C23. Works
with gcc, clang, MSVC, and any compiler that accepts the standard.

Each function emits with a profile comment:

```c
/*
 * sigmoid
 * Chain order: 1     Cost class: p1-d2-w0-c0
 * EML depth:   2     Drift risk: LOW
 * FPGA est:   0 MAC, 1 exp, 0 ln, 0 trig -> 4 cy @ 32-bit
 */
double sigmoid(double x) {
    return (1.0 / (1.0 + mg_exp((-x))));
}
```

Tuple-returning functions synthesize a `_result_t` struct:

```c
typedef struct { double e0; double e1; } park_result_t;
park_result_t park(double alpha, double beta, double theta) { ... }
```

`libmonogate.h` exposes `mg_exp`, `mg_ln`, `mg_sin`, `mg_cos`,
`mg_tan`, `mg_sqrt`, `mg_pow`, `mg_eml`, `mg_clamp`, `mg_abs`, and
the inverse / hyperbolic family. The runtime is a thin wrapper over
libm with the `mg_eml(x, y) = exp(x) - ln(y)` operator added.

### Linking into a larger project

The simplest integration: `#include "libmonogate.h"` in your project
header search path, link against `libmonogate.c`, and call the
generated functions directly. No CMake required for a single file.

For projects already using CMake, drop `software/runtime/c/` into
your tree and add it as a library target; every emitted `.c` then
becomes another library target that depends on it.

## Rust backend (`--target rust`)

```
$ eml-compile sigmoid.eml --target rust -o sigmoid.rs
$ cargo new --lib my_app && cd my_app
$ cp ../sigmoid.rs src/lib.rs
$ cargo add monogate-sys
$ cargo build && cargo clippy -D warnings
```

Output passes `cargo clippy -D warnings` cleanly on every demo.

The emitted Rust uses `f64` arithmetic; `monogate-sys` provides
`mg_exp`, `mg_ln`, etc. as `extern "C"` wrappers around libm.

```rust
/// sigmoid
/// Chain order: 1     Cost class: p1-d2-w0-c0
/// EML depth:   2     Drift risk: LOW
pub fn sigmoid(x: f64) -> f64 {
    1.0 / (1.0 + monogate_sys::mg_exp(-x))
}
```

Tuple returns map to native Rust tuples:

```rust
pub fn park(alpha: f64, beta: f64, theta: f64) -> (f64, f64) { ... }
```

## Python backend (`--target python`)

```
$ eml-compile sigmoid.eml --target python -o sigmoid.py
$ python -c "from sigmoid import sigmoid; print(sigmoid(1.0))"
0.7310585786300049
```

The generated module uses **only** `math` from the standard
library — no NumPy, no SymPy, no third-party deps:

```python
"""
Generated by EML-lang Python backend.
Source module: sigmoid
"""

from __future__ import annotations
import math

# sigmoid
# Chain order: 1    Cost class: p1-d2-w0-c0    EML depth: 2
def sigmoid(x):
    return 1.0/(1.0 + math.exp(-x))
```

Internally the backend lowers AST → SymPy → Python via the
`eml_cost.transpile.eml_tree_to_python` Tool 5 transpiler (the
same one published as part of the `eml-cost` PyPI package). When
the SymPy printer can't represent a free user-function call (e.g.
when an EML function calls another that wasn't inlined), the
backend falls back to a direct AST emitter.

This target is the canonical **golden reference** for the rest of
the toolchain — when you're debugging a numerical mismatch between
C and Verilog, compare both against the Python output first.

## LLVM IR backend (`--target llvm`)

```
$ eml-compile sigmoid.eml --target llvm -o sigmoid.ll
$ llc -filetype=obj sigmoid.ll -o sigmoid.o
$ clang sigmoid.o software/runtime/c/libmonogate.c -lm -o sigmoid.out
```

The IR is portable — same source, different `--target` flags on
`llc`:

```
$ llc -mtriple=x86_64-unknown-linux-gnu sigmoid.ll -o sigmoid.x86.s
$ llc -mtriple=aarch64-unknown-linux-gnu sigmoid.ll -o sigmoid.arm.s
$ llc -mtriple=riscv64-unknown-linux-gnu sigmoid.ll -o sigmoid.rv.s
```

Each transcendental in the EML source becomes an external call to
`libmonogate`:

```llvm
; ModuleID = 'sigmoid'
declare double @mg_exp(double)

define double @sigmoid(double %x) {
entry:
  %x.addr = alloca double
  store double %x, double* %x.addr
  %1 = load double, double* %x.addr
  %2 = fneg double %1
  %3 = call double @mg_exp(double %2)
  %4 = fadd double 1.000000e+00, %3
  %5 = fdiv double 1.000000e+00, %4
  ret double %5
}
```

The body emitter uses `alloca` / `load` / `store` for mutable
bindings; LLVM's `mem2reg` pass handles the cleanup downstream. We
intentionally don't pre-optimize — `opt -O3` on the IR is your
job.

## WebAssembly backend (`--target wasm`)

```
$ eml-compile sigmoid.eml --target wasm -o sigmoid.wasm
```

When `llc` or `clang` is on PATH, the output is wasm32 bytecode
ready to drop into a Web Audio worklet, a Workers script, or any
`WebAssembly.compile()` consumer.

When neither is on PATH, the CLI writes the LLVM IR text instead
and prints which toolchain to install:

```
wrote LLVM IR to sigmoid.wasm (no llc/clang on PATH;
install one to emit wasm bytecode)
```

The IR emitted carries `target triple = "wasm32-unknown-unknown"`,
so any wasm32-aware lowering tool produces a working module.

### Browser usage

```javascript
const { instance } = await WebAssembly.instantiateStreaming(
    fetch('sigmoid.wasm'),
);
console.log(instance.exports.sigmoid(1.0));   // 0.7310585786300049
```

The 1op.io playground demos use this path end-to-end.

## Compiling all backends at once

```
$ eml-compile sigmoid.eml --target all -o ./out
# eml-compile --target all -> ./out
  [ok]   c        ./out/sigmoid.c       (1,124 bytes)
  [ok]   rust     ./out/sigmoid.rs      (1,021 bytes)
  [ok]   lean     ./out/sigmoid.lean    (387 bytes)
  [ok]   verilog  <skipped>           (no @target(fpga) functions)
```

`--target all` writes every applicable backend's output. Hardware
targets are skipped silently when the source has no `@target(fpga)`
function.

## Optimizer control

Every software backend takes `optimize=True` by default. The CLI
mirrors this with `--no-optimize`:

```
$ eml-compile sigmoid.eml --target c --no-optimize > before.c
$ eml-compile sigmoid.eml --target c              > after.c
$ diff before.c after.c
```

The diff shows the 5-pass optimizer's effect on the emitted C —
inlined calls, constant-folded literals, hoisted CSE bindings, and
SuperBEST canonical-form rewrites. When debugging a numerical
mismatch, run with `--no-optimize` first to bisect optimizer-induced
divergences vs. genuine bugs.

See [`architecture/optimizer_pipeline.md`](architecture/optimizer_pipeline.md)
for the per-pass detail.

## Cross-backend equivalence (Patent #22)

The harness at `tools/equivalence/cross_target_check()` compiles a
chosen function through Python, C, Rust, and Lean; runs the same
input vectors through each; and asserts agreement within 1e-12.
Toolchain-missing backends report `available=False` rather than
failing the run.

```python
from tools.equivalence import cross_target_check

report = cross_target_check(
    "lang/spec/stdlib/math.eml",
    "sigmoid",
    [(0.0,), (1.0,), (-1.0,)],
    targets=("python", "c", "rust"),
)
print(report.render())
```

Output:

```
Equivalence report for 'sigmoid' (3 vectors):
  python  : max abs err 0e+00, max rel err 0e+00
  c       : max abs err 1e-16, max rel err 2e-16
  rust    : max abs err 1e-16, max rel err 2e-16
  overall:  MATCH
```

Run it routinely in CI; the agreement is a load-bearing property
of the architecture, not a side effect.
